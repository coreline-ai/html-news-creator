## 개발 목적

`https://github.com/coreline-ai/multi_model_tui.git`의 로컬 프록시를 활용해 `html-news-creator`의 AI 호출 계층을 OpenAI API 호환 방식으로 사용할 수 있게 만드는 계획을 정의한다.

핵심 목표는 다음과 같다.

- `html-news-creator`는 OpenAI SDK/Responses API 호환 클라이언트만 바라본다.
- 실제 로컬 실행에서는 `multi-model-tui` 프록시가 `/openai/v1/responses`와 `/openai/v1/chat/completions`를 제공한다.
- `multi-model-tui`의 Codex provider는 Codex CLI 인증 파일(`~/.codex/auth.json`)을 사용해 Codex direct transport를 수행한다.
- 필요 시 Codex CLI 자체도 `multi-model-tui`를 custom model provider로 바라볼 수 있도록 `config.toml` 설정 방안을 검증한다.

## 개발 범위

- `multi-model-tui`를 로컬 sidecar proxy로 사용하는 실행/연동 계획 수립
- OpenAI API 호환 base URL, 모델, 인증 환경변수 계약 정의
- `html-news-creator` 내부 AI client 추상화 방향 정의
- `/openai/v1/responses` 우선, `/openai/v1/chat/completions` fallback 전략 정의
- Codex CLI `config.toml` custom provider 연동 검증 계획 수립
- Codex CLI 내장 `$imagegen` 스킬을 통한 `gpt-image-2` 기반 이미지 생성/편집 워크플로우 정의
- 프로젝트 이미지 산출물의 저장 위치, 파일명 규칙, 검수 기준 정의
- 호환성 테스트 케이스 정의

## 제외 범위

- 이번 문서 작성 단계에서는 실제 코드 구현을 하지 않는다.
- `multi-model-tui` 저장소를 즉시 vendoring/submodule로 추가하지 않는다.
- OpenAI 공식 API를 대체한다고 보장하지 않는다.
- Codex/ChatGPT 비공개 backend API 안정성을 운영 SLA로 간주하지 않는다.
- Claude/Gemini provider 고급 기능까지 `html-news-creator` 요구사항으로 확장하지 않는다.
- 이미지 생성 기본 경로에서는 `gpt-image-1.5` 또는 별도 이미지 모델 fallback을 사용하지 않는다.
- `gpt-image-2`가 충분하므로 네이티브 투명 배경이 필요한 특수 케이스는 기본 범위에서 제외한다.

## 문서 기반 제약 사항

- OpenAI 공식 문서 기준으로 Codex custom model provider는 `model_providers.<id>.base_url`, `env_key`, `wire_api = "responses"` 형태를 지원한다.
- Codex custom provider의 `wire_api`는 Responses API만 사용하는 것으로 문서화되어 있으므로, Codex CLI 연동 검증은 `/openai/v1/responses`를 기준으로 한다.
- OpenAI 공식 문서는 Responses API를 Chat Completions API의 상위 집합으로 설명하고, 신규/추론 모델 흐름은 Responses API로 점진 이전할 것을 권장한다.
- Responses API와 Chat Completions API는 structured outputs, function calling, state handling 형태가 다르므로, 호환 프록시 테스트는 두 API를 분리해서 검증한다.
- `multi-model-tui` 현재 구조는 이미 다음 호환 엔드포인트를 가진다.
  - `POST /openai/v1/chat/completions`
  - `POST /openai/v1/responses`
- `multi-model-tui` Codex provider는 기본적으로 `~/.codex/auth.json`을 읽고, fallback으로 `~/.chatgpt-codex-proxy/tokens.json`을 사용한다.
- Codex CLI는 이미지 생성/편집을 직접 지원하며, 명시 호출은 프롬프트에 `$imagegen`을 포함한다.
- Codex CLI 내장 이미지 생성은 `gpt-image-2`를 사용하므로, 본 프로젝트의 리포트 히어로 이미지, 썸네일, 아이콘 초안, placeholder art에는 `gpt-image-2`를 표준으로 고정한다.
- `gpt-image-2`는 투명 배경 `background: "transparent"`를 현재 지원하지 않으므로, 투명 PNG가 필요하면 chroma-key 배경 생성 후 로컬 제거 방식으로 처리하고 모델 fallback은 기본 계획에 포함하지 않는다.

## 목표 아키텍처

```text
html-news-creator backend
  -> OpenAI-compatible AI Gateway Client
     base_url=http://127.0.0.1:4317/openai/v1
     endpoint=/responses 우선
  -> multi-model-tui local proxy
     /openai/v1/responses
     /openai/v1/chat/completions
  -> provider router
  -> codex provider
  -> Codex CLI auth file (~/.codex/auth.json)
  -> Codex direct transport
```

선택적 역방향 검증:

```text
Codex CLI
  -> model_provider = multi_model_tui
  -> base_url=http://127.0.0.1:4317/openai/v1
  -> /responses
  -> multi-model-tui local proxy
```

## Phase 2. Local proxy 실행 계약 정의

### 목표

`html-news-creator`에서 로컬 OpenAI 호환 proxy를 안정적으로 호출할 수 있도록 실행 방식과 환경변수를 고정한다.

### 구현 태스크

- [ ] sidecar 실행 방식을 결정한다: 외부 clone 우선, repo vendoring은 후순위
- [ ] 로컬 프록시 기본 주소를 `http://127.0.0.1:4317`로 고정한다.
- [ ] OpenAI 호환 base URL을 `http://127.0.0.1:4317/openai/v1`로 정의한다.
- [ ] `OPENAI_BASE_URL` 또는 `AI_GATEWAY_BASE_URL` 중 최종 환경변수명을 결정한다.
- [ ] `OPENAI_API_KEY`는 OpenAI SDK 호환을 위한 dummy 값 또는 proxy 전용 키로 분리한다.
- [ ] proxy 자체 인증이 필요하면 `MULTI_MODEL_TUI_API_KEY`를 추가하는 방향을 검토한다.
- [ ] Codex 인증 선행 조건을 문서화한다: `codex login` 또는 기존 `~/.codex/auth.json` 존재.
- [ ] health check 실패 시 공식 OpenAI API로 fallback할지 fail-fast할지 결정한다.
- [ ] local proxy 모드에서는 `OPENAI_API_KEY`를 사용자에게 요구하지 않도록 `Settings` validation 정책을 분리한다.
- [ ] `/embeddings`처럼 local proxy가 아직 지원하지 않는 endpoint는 `EMBEDDING_PROVIDER=local-hashing` fallback을 기본값으로 두고, `EMBEDDING_PROVIDER=openai`일 때만 `OPENAI_API_KEY` 필요 조건을 표시한다.

### 권장 환경변수 초안

```env
AI_PROVIDER=openai-compatible
AI_GATEWAY_BASE_URL=http://127.0.0.1:4317/openai/v1
AI_GATEWAY_API_KEY=local-dev-only
AI_GATEWAY_MODEL=gpt-5
AI_GATEWAY_PREFER_RESPONSES=true
AI_GATEWAY_TIMEOUT_MS=60000
```

주의:

- `AI_PROVIDER=openai-compatible` + `AI_GATEWAY_BASE_URL` 경로에서는 `OPENAI_API_KEY`가 필수가 아니다.
- 이 경로의 bearer 값은 OpenAI SDK 호환용 dummy/proxy key일 수 있으며, 실제 Codex 인증은 `~/.codex/auth.json`에 의존한다.
- 반대로 `AI_PROVIDER=openai`, `EMBEDDING_PROVIDER=openai`, Image API batch 등 proxy 밖의 공식 OpenAI endpoint를 직접 호출하는 단계는 `OPENAI_API_KEY`가 필요하다.
- Phase 4 embedding 포함 full pipeline도 기본값 `EMBEDDING_PROVIDER=local-hashing`이면 `OPENAI_API_KEY` 없이 실행 가능하도록 설계한다.

- [ ] Codex CLI가 local proxy로 요청한 payload가 `/openai/v1/responses`에서 정상 파싱되는지 확인한다.
- [ ] Codex CLI → proxy → Codex direct transport 구조에서 요청 루프가 발생하지 않는지 확인한다.

### Codex CLI 설정 초안

```toml
model = "gpt-5"
model_provider = "multi_model_tui"

[model_providers.multi_model_tui]
name = "Multi Model TUI Local Proxy"
base_url = "http://127.0.0.1:4317/openai/v1"
env_key = "MULTI_MODEL_TUI_API_KEY"
wire_api = "responses"
request_max_retries = 2
stream_max_retries = 2
stream_idle_timeout_ms = 300000
```

주의:

- 위 설정은 검증용 초안이다.
- `multi-model-tui`가 별도 API key 검증을 하지 않는다면 `env_key`는 dummy bearer 용도로만 동작할 수 있다.
- Codex CLI custom provider는 공식 문서상 Responses wire API 기준이므로 Chat Completions 호환 검증과 분리한다.

## Phase 6. Codex CLI `gpt-image-2` 이미지 생성 워크플로우 정의

### 목표

Codex CLI에서 내장 `$imagegen` 스킬과 `gpt-image-2`를 사용해 `html-news-creator`의 이미지 자산을 생성하고, 프로젝트 내부에서 재현 가능한 산출물 관리 규칙을 고정한다.

이번 계획에서는 `gpt-image-2`로 충분하다고 판단하며, 이미지 생성 기본 경로는 추가 모델 fallback 없이 `$imagegen` 기반으로 단순화한다.

### 적용 원칙

- 기본 호출 방식은 Codex CLI 프롬프트에 `$imagegen`을 명시하는 방식으로 한다.
- 기본 모델은 Codex CLI 내장 이미지 생성의 `gpt-image-2`로 고정한다.
- 리포트 히어로 이미지, 뉴스 썸네일, 섹션 일러스트, placeholder 이미지, 앱/문서용 비주얼 초안에 우선 적용한다.
- 프로젝트에서 참조하는 최종 이미지는 Codex 기본 생성 위치가 아니라 workspace 내부에 복사/이동한다.
- 텍스트가 포함된 이미지 생성은 오탈자 리스크가 있으므로 가능하면 이미지 안의 텍스트를 최소화한다.
- 투명 배경이 필요하면 `gpt-image-2` 네이티브 투명 배경을 기대하지 않고, 단색 chroma-key 배경 생성 후 로컬 제거를 후속 작업으로 둔다.

### 권장 저장 위치

프로젝트 구조 확정 전에는 `assets/images/generated/`를 기본 후보로 사용하고, 웹에서 직접 서빙해야 하는 경우만 `public/images/generated/`를 사용한다.

## Coreline Agent OAuth Proxy 추가 검토 결과

`coreline-agent`도 Codex CLI OAuth 기반 OpenAI-compatible proxy 후보로 검증했다. `codex-backend` provider는 `~/.codex/auth.json` 또는 `CODEX_AUTH_PATH`를 사용하며, `coreline-agent-proxy`는 `/openai/v1/responses`와 `/openai/v1/chat/completions`를 제공한다.

권장 검증 경로는 다음과 같다.

```text
Codex CLI
  -> model_provider = coreline_agent
  -> base_url=http://127.0.0.1:4317/openai/v1
  -> /responses
  -> coreline-agent-proxy
  -> codex-backend
  -> ~/.codex/auth.json
  -> chatgpt.com/backend-api/codex/responses
```

주의: Codex CLI가 proxy를 호출하는 구조에서는 proxy 기본 provider를 `codex-cli`로 두지 말고 `codex-backend`로 둔다. 그렇지 않으면 `Codex CLI -> proxy -> codex-cli -> Codex CLI` 재귀 루프 위험이 있다. 상세 내용은 [Coreline Agent Codex CLI OAuth Proxy 검토](./review_20260427_coreline_agent_oauth_proxy.md)를 따른다.

## 잔여 리스크 / 후속 과제

- `multi-model-tui`의 Codex direct transport는 공식 OpenAI API가 아니라 Codex/ChatGPT backend 형태에 의존하므로 장기 안정성 리스크가 있다.
- OpenAI SDK가 기대하는 Responses API의 모든 세부 이벤트/필드를 local proxy가 완전하게 지원하는지 검증이 필요하다.
- `/v1/models` 같은 보조 OpenAI API endpoint가 필요할 수 있다.
- `previous_response_id`, hosted tools, file input, citations, caching은 현재 partial/unverified 영역으로 분리 관리해야 한다.
- `gpt-image-2`는 투명 배경을 직접 지원하지 않으므로, 투명 PNG는 chroma-key 제거 품질 검수가 필요하다.
- 이미지 안의 텍스트 렌더링은 오탈자/왜곡 가능성이 있으므로 리포트 타이틀·로고·정확한 문구는 HTML/CSS 레이어로 처리하는 것이 안전하다.
- 운영 배포에서는 local proxy sidecar보다 공식 OpenAI API 사용 경로를 우선 검토한다.

## 최종 결과 요약

- 이 문서는 `multi-model-tui`를 OpenAI API 호환 local proxy로 사용하기 위한 구현 전 계획이다.
- 구현 우선순위는 `Responses API 호환 → html-news-creator AI Gateway → Codex CLI custom provider 검증 → 고급 기능 확장` 순서다.
- Codex CLI 이미지 생성은 내장 `$imagegen`과 `gpt-image-2`를 표준으로 사용하며, 별도 이미지 모델 fallback은 기본 계획에서 제외한다.
- 현재 단계에서는 문서만 추가했으며 실제 코드 변경은 하지 않았다.
