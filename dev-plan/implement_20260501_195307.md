## 개발 목적

- 프롬프트 인젝션, HTML sanitizer fallback, 잘못된 정책 override 등 보안·신뢰성 리스크를 줄인다.
- 편집 점수 공식과 클러스터 선정 로직의 단일 출처를 마련해 정책 변경 회귀를 줄인다.
- 일일 파이프라인의 운영 실패를 정확히 감지하고, KST 날짜·소스 필터·렌더 통계·이미지 fallback 정합성을 맞춘다.
- Prometheus 기반 최소 운영 메트릭과 부분 파이프라인 통합 테스트를 추가한다.
- README/AGENTS/운영 문서가 현재 코드와 어긋나지 않도록 정리한다.

## 공통 진행 규칙

- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.
- 구현 중 발생한 이슈는 해당 Phase에서 수정하고 기록한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.
- 별도 브랜치나 PR을 만들지 않고 `main` 브랜치에서만 작업한다.
- 각 Phase 완료 시점마다 테스트를 통과시킨 뒤 변경 내용을 작게 묶어 기록한다.
- 보안 관련 변경은 실패 시 조용히 fallback하지 말고 명시적으로 fail-fast 또는 4xx/5xx를 반환한다.
- 운영 메트릭은 실제로 `/metrics`에서 관측 가능한 경로만 완료로 본다.

### 구현 태스크

## Phase 5. 운영 메트릭 + timeout 설정 검증

### 목표

- run 상태와 주요 latency를 `/metrics`에서 관측 가능하게 한다.
- run timeout 기본 max runtime은 이미 `Settings.admin_run_default_max_runtime_sec`가 단일 출처이고, `run_runner.py`의 `_DEFAULT_MAX_RUNTIME_SEC`는 단순 alias이므로 현 구조를 유지한다.
- 이번 Phase에서는 timeout 기본값 경로와 override 동작을 회귀 테스트로 고정하고, `_GRACE_PERIOD_SEC`/`_MAX_LINES_PER_RUN`의 Settings 이관은 별도 요구가 없으면 범위에서 제외한다.

### 구현 태스크

- [ ] `requirements.txt`에 `prometheus-client` 추가.
- [ ] `app/metrics.py` 신규 생성: `runs_total`, `run_duration_seconds`, `active_runs`, `llm_latency_seconds` 정의.
- [ ] `app/admin/api.py`에 `GET /metrics` 엔드포인트 추가. Prometheus text exposition format 반환.
- [ ] `app/admin/run_runner.py`에서 run 시작/완료/실패/timeout 시 `runs_total`, `run_duration_seconds`, `active_runs` 갱신.

## main 단일 브랜치 작업 방식

- 별도 브랜치와 PR은 만들지 않는다. 모든 작업은 `main` 브랜치에서만 진행한다.
- Phase 순서를 유지하며, 한 Phase의 자체 테스트가 끝나기 전 다음 Phase 구현을 시작하지 않는다.
- 각 Phase는 가능한 한 작은 변경 단위로 나누고, 테스트 결과를 사용자에게 보고한 뒤 다음 Phase로 진행한다.
- 중간 저장이 필요하면 PR 대신 `main`에서 직접 커밋하되, 커밋 전 최소 해당 Phase 테스트를 통과시킨다.
- 실패하거나 범위가 커지는 항목은 즉시 중단하고 이 문서의 `이슈 및 수정`에 기록한 뒤 사용자에게 보고한다.

## 잔여 리스크 / 후속 과제

- Prometheus LLM latency는 subprocess 경계를 넘는 설계가 필요하다. parent process가 child JSON log를 파싱해 observe하는 방식이 단순하지만, 로그 포맷 변경에 취약하다. multiprocess mode는 운영 설정이 복잡하므로 1차 구현에서는 피한다.
- HDBSCAN noise singleton 승격은 리포트 섹션 수와 topic quota에 영향을 줄 수 있으므로 selection 테스트 기대값 변경이 필요할 수 있다.
- true dry-run을 구현하려면 단계 간 in-memory artifact 전달이 필요해 범위가 크다. 이번 계획에서는 문서화 또는 명확한 명칭 조정을 우선한다.
- `from __future__ import annotations` 전면 적용은 파일 수가 많아 별도 cleanup 계획으로 분리하는 것이 안전하다.
