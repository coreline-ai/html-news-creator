## 개발 목적
`안 ①-R` 이후 단계로 Run 시작/종료 상태를 `job_runs` DB 테이블에 최소 영속화한다. 서버 재시작 후에도 완료/실패된 Run의 요약 상태를 조회할 수 있게 하되, 진행 로그 전체 저장과 subprocess 재연결은 제외한다.

## 개발 범위
- Run 상태를 `JobRun` row로 best-effort 생성/갱신한다.
- `run_id`와 `JobRun.id`를 같은 UUID로 맞춰 `/api/runs/{run_id}` DB fallback 조회가 가능하게 한다.
- `/api/runs`, `/api/runs/{run_id}`가 메모리 Run과 DB JobRun을 병합/조회한다.
- DB-only running/queued Run은 `trackable=false`로 반환해 프론트가 stale 상태로 안내한다.
- 프론트 `GlobalRunIndicator`가 DB-only stale Run을 SSE 구독하지 않고 안내 후 정리한다.
- 관련 backend/frontend 테스트와 E2E mock을 보강한다.

## 제외 범위
- `JobLog`에 stdout/stderr 전체 로그 저장.
- 진행 이벤트마다 DB update하는 full progress persistence.
- 서버 재시작 후 살아있는 subprocess에 재연결.
- multi-worker run orchestration.
- 신규 DB 마이그레이션. 기존 `job_runs` 스키마만 사용한다.
