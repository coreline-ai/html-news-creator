# Implementation Plan: AI Trend Report Engine

> RSS·GitHub·arXiv·웹을 수집→분류→클러스터→검증→렌더링→배포하는 한국어 AI 트렌드 리포트 자동화 엔진 구축
> Generated: 2026-04-27
> Project: html-news-creator (ai-trend-report-engine)

---

## 1. Context

### 1.1 Why
매일 수백 개의 AI 관련 공식 발표·논문·GitHub 릴리즈·소셜 신호를 수작업으로 수집·요약하는 것은 비효율적이다. 공식 출처 검증 없이 소셜 루머를 사실처럼 게재하면 신뢰도가 훼손된다. 10단계 자동 파이프라인으로 수집→검증→렌더링→배포를 완전 자동화하고, `fact / social_signal / inference / unknown` 4가지 클레임 타입을 분리하여 신뢰 가능한 한국어 AI 트렌드 리포트를 매일 자동 발행한다.

### 1.2 Current State
- `docs/` 안에 PRD/TRD/스키마/프롬프트/런북 문서 존재
- `schema.sql` 완성 (14개 테이블, pgvector 포함)
- `docker-compose.yml` 완성 (postgres/redis/minio)
- `.env.example` 완성
- `github-actions-daily-report.yml` 완성
- **소스코드 0%** — `app/`, `scripts/`, `templates/`, `tests/` 미존재

### 1.3 Target State
- `python scripts/run_daily.py --mode full` 한 명령으로 10단계 파이프라인 실행
- RSS + GitHub + arXiv 3개 공식 소스에서 매일 자동 수집
- LLM 관련도 분류 → 공식 출처 검증 → 클러스터링 → 섹션 요약
- Jinja2 HTML + Playwright PNG/PDF 아티팩트 생성
- `public/news/YYYY-MM-DD-trend.html` Netlify 자동 배포
- GitHub Actions cron `0 22 * * *` (07:00 KST) 완전 자동화

### 1.4 Scope Boundary
**In scope:**
- Phase 0: 프로젝트 초기화, DB 마이그레이션
- Phase 1: RSS/GitHub/arXiv 수집 + Firecrawl→Crawl4AI→Trafilatura 추출 + LLM 분류 + Jinja2 렌더링 (MVP)
- Phase 2: 공식 출처 검증 + 신뢰도 스코어
- Phase 3: 이미지 다운로드 + PaddleOCR/Surya OCR + 비전 캡션
- Phase 4: 임베딩 + HDBSCAN 클러스터링 + 트렌드 점수
- Phase 5: 생성형 HTML + HTML sanitizer + Playwright visual regression
- Phase 6: GitHub Actions CI/CD + Netlify 배포 자동화
- Phase 7: X/Twitter 수집 + FastAPI 관리자 대시보드

**Out of scope:**
- 사용자 인증/구독/결제
- 실시간 스트리밍
- 다국어 지원 (한국어 전용)
- Vercel 배포 (Netlify 우선)

### 2.2 Key Design Decisions

| 결정 사항 | 선택 | 근거 |
|-----------|------|------|
| 패키지 관리 | pip + requirements.txt | GitHub Actions 호환성, 단순성 |
| DB | PostgreSQL + pgvector (pgvector/pgvector:pg16) | 임베딩 클러스터링, 14개 테이블 스키마 완성 |
| 캐시/큐 | Redis 7 | job 상태 관리, rate limit 제어 |
| 스토리지 | MinIO (로컬) / S3-compatible (운영) | 이미지·아티팩트 보관 |
| 크롤러 체인 | Firecrawl → Crawl4AI → Trafilatura | 유료 우선, 오픈소스 폴백 |
| LLM | OpenAI gpt-5.5 (분류/요약/검증) | 문서 명시 모델 |
| 임베딩 | text-embedding-3-small (1536차원) | 스키마 `vector(1536)` 일치 |
| 클러스터링 | HDBSCAN + cosine similarity | 가변 클러스터 수, 노이즈 처리 |
| 렌더링 | Jinja2 (고정) + LLM 생성형 HTML (확장) | MVP는 Jinja2, Phase 5에서 생성형 추가 |
| 진입점 | `scripts/run_daily.py` | `--from-step`, `--to-step` 부분 재실행 지원 |
| 클레임 분리 | fact / social_signal / inference / unknown | 공식 출처 없는 주장 사실 처리 금지 |

## 6. Edge Cases & Risks

| 위험 요소 | 영향도 | 완화 방안 |
|-----------|--------|-----------|
| Firecrawl 과금 초과 | 높음 | Crawl4AI/Trafilatura 폴백, 일일 요청 한도 Redis 카운터 |
| OpenAI rate limit | 높음 | tenacity exponential backoff, prompt caching 활성화 |
| LLM JSON 파싱 실패 | 중간 | repair prompt 재시도 → 실패 시 deterministic fallback → human review queue |
| PaddleOCR 한국어 인식 낮음 | 중간 | `lang="korean"` 파라미터, confidence < 0.6 시 Surya 폴백 |
| Playwright 메모리 누수 | 중간 | 브라우저 인스턴스 재사용 풀 max 2, timeout 30초 강제 종료 |
| DB pgvector 임베딩 저장 실패 | 중간 | `CREATE EXTENSION vector` 확인, pgvector 버전 0.7.0+ 필요 |
| HTML injection (크롤링 콘텐츠) | 높음 | Jinja2 autoescape=True, bleach sanitizer 필수 |
| robots.txt 위반 | 중간 | `robotparser` 크롤링 전 체크, 위반 URL skip |
| GitHub Actions 90분 타임아웃 | 중간 | Phase 1~4 병렬화, 단계별 캐싱, timeout-minutes: 90 설정 |
| X API 약관 위반 | 낮음 | `X_ENABLED=false` 기본값, 공식 API 우선, 정책 주기적 검토 |
| secret 로그 노출 | 높음 | structlog secret_masker, `OPENAI_API_KEY` 패턴 마스킹 |
| 가짜 뉴스 확산 | 높음 | official verification 배지, confidence 표시, unverified 명시 |
| Supabase 무료 플랜 500MB 한도 | 낮음 | raw_items 180일, job_logs 30일 자동 삭제 (`pg_cron` 또는 cron job) |

## 7. Execution Rules

1. **독립 모듈**: 각 Phase는 독립적으로 구현하고 테스트한다
2. **완료 조건**: 모든 태스크 체크박스 + 모든 테스트 케이스 통과 후 다음 Phase
3. **테스트 실패 워크플로우**: 에러 분석 → 근본 원인 수정 → 재테스트 → 통과 후 진행
4. **병렬 실행**: Phase 2와 Phase 3은 동시 진행 가능
5. **커밋 단위**: Phase 완료 시 1개 커밋, 메시지: `feat(phase-N): <설명>`
6. **보안 금지 영역**:
   - API 키 하드코딩 절대 금지
   - Jinja2 `autoescape=False` 사용 금지
   - SSRF 체크 없는 외부 URL fetch 금지
   - `service_role` 키 클라이언트 노출 금지
7. **비용 절약**: OpenAI 호출은 mock 우선, `-m integration` 마커로 명시적 실행
8. **운영 지표 목표** (`11_OBSERVABILITY.md` 기준):
   - daily job success rate ≥ 95%
   - report generation time ≤ 30분
   - official verification coverage ≥ 70%
   - unverified claim rate ≤ 10%
   - render success rate ≥ 99%
9. **릴리즈 게이트** (`08_TEST_PLAN.md` §10):
   - unit test 90% 이상 통과
   - integration test 통과
   - sample daily report 생성 성공
   - visual screenshot 통과
   - schema migration 적용 성공
   - secrets 문서화
   - rollback 절차 문서화
